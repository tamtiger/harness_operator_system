**Session:** Fixed bug where file uploads >10MB crashed with 500 error on AWS Lambda

## Example Questions & Answers

### Q1. What was the root cause of the bug, and what conditions allowed it to stay hidden?

**A1:** Root cause was synchronous file buffering in multer middleware. The handler called `req.file.buffer` which loaded the entire file into Lambda's 128MB memory limit. This stayed hidden because: (1) local testing with small files (<10MB) never triggered it, (2) no production monitoring on memory usage, (3) no test for files near the Lambda limit.

---

### Q2. Why did we choose `busboy` over alternatives like `formidable`, `businneds`, or raw stream handling?

**A2:** Trade-off comparison:
- `busboy`: Small footprint, handles backpressure natively, battle-tested in Node ecosystem
- `formidable`: Heavier, more features (file rename, cleanup) we didn't need
- Raw streams: Would require custom error handling, slower to implement
- `business`: Not actively maintained

We chose `busboy` for speed-to-fix and reliability. We could revisit if performance becomes the bottleneck (unlikely given current throughput).

---

### Q3. Draw the execution flow from request entry to S3 write. What happens if S3 upload fails mid-stream?

**A3:**
```
POST /upload 
  → multer middleware 
    → busboy.on('file', chunk => {...})
      → S3.putObject(chunk)
        → success: continue to next chunk
        → error: emit error event
          → busboy.destroy()
          → res.status(500).send('Upload failed')
```

**If S3 fails mid-stream:** multer catches S3 error, destroys the busboy stream, and sends 500 response. Incomplete file stays in S3 bucket (not cleaned up). **Improvement:** Add S3 cleanup on error (call `deleteObject` if upload fails).

---

### Q4. What happens if a user uploads a 500MB file? Will the fix handle it, or are there new limits?

**A4:** The fix handles 500MB fine because we stream to S3 (no memory buffering). However, there are new limits:
- **S3 per-chunk size:** Default 5MB chunks per `putObject`, ok for 500MB (100 chunks)
- **Request timeout:** Lambda has 15-minute timeout by default, should be enough for streaming
- **Network bandwidth:** AWS Lambda egress is charged per GB, 500MB = $0.02 cost per upload

The streaming approach scales, but we should add test for large files and monitor S3 timeout behavior.

---

### Q5. If we need to support file rename on upload, what would break?

**A5:** Current flow assumes filename from `Content-Disposition` header. To support rename:
- **Would break:** Direct S3 write with original filename
- **Would need:** Capture rename intent (POST param? header?), transform S3 key before `putObject`
- **Safe approach:** Don't rename during stream (too late to undo). Instead, upload with temp name, then `copyObject` + `deleteObject` after stream completes.

---

### Q6. Why not use AWS Lambda's native request streaming instead of `busboy`?

**A6:** Lambda doesn't expose raw Node.js stream from `event.body`. Instead, it always buffers the body into a string/buffer. To get true streaming in Lambda, you'd need HTTP API (not REST API) or ALB integration, both requiring infrastructure changes. `busboy` + `req` stream is the simplest for current setup.

### Q8. If S3 chunking fails on the 3rd chunk of 5, is the incomplete file cleaned up?

**A8:** **No, currently not cleaned up.** The incomplete file remains in S3 as a partial object. To fix this:
1. **Abort multipart upload:** Use `abortMultipartUpload` API before destroying the stream
2. **Mark incomplete:** Add metadata tag `upload-status: incomplete` so cron jobs can clean up later

Current code doesn't do either, so there's data leak. **Action item:** Add S3 cleanup in error handler.

---

### Q9. Why is `busboy` better for memory than buffering, given both ultimately read the file?

**A9:** 
- **Buffering:** Read entire 100MB into RAM before writing anywhere (peak memory: 100MB)
- **Streaming:** Read 5MB chunk, write to S3, discard chunk, repeat (peak memory: 5MB + overhead)

Result: Streaming reduces peak memory by 20×. For Lambda with 128MB limit, buffering maxes out at ~128MB file, streaming handles GB+ files.

- **Metric 3:** HTTP 500 error rate on `/upload` (should drop to 0.1% baseline)
- **Log analysis:** Search for `busboy` stream errors in CloudWatch logs for first 24h
- **Canary:** Deploy to 10% of traffic first, monitor metrics for 1 hour before full rollout

If any metric deviates, roll back immediately and investigate.

**Problem:** Buffering large files in Lambda crashes because memory is limited.

**Solution:** Use streaming parser (busboy) to process files in chunks, write to S3 as streams.

**Key insight:** For I/O-bound operations (file upload), always stream when memory is constrained. This scales to GB+ files with minimal memory overhead.

**Lessons for next time:**

- Add monitoring on memory usage by endpoint
