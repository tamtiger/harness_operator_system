# Issue Triage & Management

Manage and prioritize issues or tasks using a structured state machine. Ensure all triage comments start with:
```
> *This was generated by AI during triage.*
```

## Roles & States

### Categories (Exactly one per issue)
- `bug` — Something is broken, throwing errors, or behaving incorrectly.
- `enhancement` — A new feature, improvement, or optimization.

### States (Exactly one per issue)
- `needs-triage` — New issue, awaiting evaluation by maintainers.
- `needs-info` — Awaiting additional details (logs, reproduction steps) from the reporter.
- `ready-for-agent` — Fully specified, clear acceptance criteria, ready for autonomous execution.
- `ready-for-human` — Requires human decision-making, design choices, or high-risk implementation.
- `wontfix` — Closed, out of scope, or will not be actioned.

## State Transitions
1. Unlabeled/new issues start at `needs-triage`.
2. Move to `needs-info` if requirements are fuzzy or reproduction is missing.
3. Move to `ready-for-agent` or `ready-for-human` once fully understood.
4. If transitioning to `ready-for-agent`, you **must** generate and attach a durable **Agent Brief**.

## Writing Agent Briefs for `ready-for-agent`
When an issue is marked `ready-for-agent`, it becomes the contract for the AFK agent. It must be durable (file paths and line numbers can change, so describe interfaces and contracts instead).

Use the following template:

```markdown
## Agent Brief

**Category:** bug / enhancement
**Summary:** One-line description of the goal

**Current Behavior:**
What happens now (broken behavior or status quo).

**Desired Behavior:**
Detail how the system should behave after completion, including edge cases and error handling.

**Key Interfaces:**
- `InterfaceName` / `functionSignature()` contracts.
- Configuration and schema modifications.

**Acceptance Criteria:**
- [ ] Criterion 1 (verifiable and testable).
- [ ] Criterion 2.

**Out of Scope:**
- Items explicitly excluded from this task.
```
